# #SMS (Shutri Media Solution) - Technical Specification

## 1. Overview

### 1.1. Core Concept

**#SMS** is a professional media workflow engine designed to bridge the gap between AI-native content generation (LLMs) and multi-platform publishing. It operates as a modular pipeline that ingests raw assets, "massages" them into platform-specific formats, and publishes them to various destinations.

The system is built on an **Ontology-First** approach, where the "Episode" or "Chapter" number serves as the primary key (the "Master Key") that binds text, images, and video assets together.

### 1.2. Core Philosophy: The AI-Native Manifesto

#SMS is built as an **AI-Native Application**, governed by the following principles:

1.  **Code is Free:** The software is open-source and free of charge. Users are not charged for the logic that orchestrates their media pipeline.
2.  **Decoupled AI Costs:** While #SMS optionally integrates with AI (e.g., Gemini via n8n), the user pays the AI provider directly for their usage. The application does not act as a middleman or "wrap" AI for profit.
3.  **Natural Intelligence Fallback:** The user is always free to use their own "Natural Intelligence" to edit and format content. The software provides the tools for automation, but the AI is never mandatory.
4.  **Zero Responsibility:** As an open-source tool, we take no responsibility for the behavior of the code, the outputs generated by integrated AI models, or the consequences of publishing said content. The user remains the ultimate authority and responsible party for their media assets.

### 1.3. Key Terminology

*   **Master Key (MK):** The unique numerical identifier for a piece of content (e.g., `241`).
*   **Asset:** Any raw file (Text, Image, Video) ingested into the system.
*   **Ingestion:** The process of capturing raw assets and storing them in an organized MK-indexed directory.
*   **Massage:** The transformation layer where raw content is sanitized, formatted, and enriched for a specific target (e.g., KaTeX hardening for mdBook, character-limit pruning for LinkedIn).
*   **Platform Target:** The specific destination format (e.g., `mdbook`, `linkedin`, `nostr`).
*   **Publish:** The final stage where formatted content is committed to a repository (Git) or pushed to an API.

## 2. System Architecture

The application is structured into three primary functional modules:

### 2.1. Module I: Ingest
*   **Responsibility:** Asset capture and storage.
*   **Action:** Creates a numbered folder (`/assets/{MK}/`) and moves/renames incoming files to match the MK.
*   **Input Types:** `.rs` (shielded text), `.png/.jpg` (images), `.mp4` (video).

### 2.2. Module II: Massage
*   **Responsibility:** Transformation and Formatting.
*   **Action:** 
    *   Strips "Shields" from text.
    *   Applies regex-based sanitization (KaTeX, Footnotes).
    *   Injects multimedia snippets (Video Carousels, Social Links).
    *   Adapts content length and metadata for different platform targets.

### Milestone 6: Target Deployment (Local Publish)
*   **Responsibility:** Delivering the massaged content to the local target repository.
*   **Action:** 
    *   Automatically migrates the final assets from the #SMS workspace to the specified local path of the target repository (e.g., the `src/` folder of a project).
    *   Ensures directory structure (images, videos) is preserved in the destination.
    *   *Note: Git operations (Commit/Push) are out of scope for this version.*

## 4. Development Phases (Milestones)

### Phase 1: Prototype (The Foundation)
*   **Milestone 1: Project Setup & CLI Skeleton.** Basic project structure and command parsing (`sms ingest --mk 241`).
*   **Milestone 2: Basic Ingest.** Creating MK folders and migrating files from a local source (Downloads folder).
*   **Milestone 3: The mdBook Massage.** Implementing the core sanitization logic inherited from `mdIngest` (KaTeX, Title limits).

### Phase 2: Multi-Platform (The Expansion)
*   **Milestone 4: Image & Video Ingest.** Handling multimedia assets and generating the cinematic scroll logic.
*   **Milestone 5: Multi-Target Massage.** Adding logic for `linkedin` and `nostr` targets.
*   **Milestone 6: Target Deployment (Local Publish).** Automating the migration of massaged content to the target repository folder.

### Phase 3: Automation (The n8n Layer)
*   **Milestone 7: Webhook/API Interface.** Exposing `sms` functionality for integration with n8n workflows.
*   **Milestone 8: Full Automation Test.** End-to-end run from LLM export to multi-platform delivery.

## 5. Invocation and Options

*   `sms ingest --mk <MK> --path <SOURCE>`
*   `sms massage --mk <MK> --target <mdbook|linkedin|nostr>`
*   `sms publish --mk <MK> --target <mdbook>`

## 6. Testing & Approval Strategy

### 6.1. Tiered Approval Process
To ensure development rigor, the project follows a two-tiered testing and approval workflow:

1.  **Engine Tier (Phases 1 & 2): CLI-Based Verification.** 
    *   Approvals are granted based on the performance and output of the **Rust CLI Engine**.
    *   The user verifies successful asset migration, directory creation, and text sanitization directly via the terminal and file system.
2.  **Cockpit Tier (Phase 3): n8n UI-Based Verification.**
    *   Approvals are granted based on the end-to-end performance of the **n8n Workflow**.
    *   The user verifies success via the n8n UI on `localhost`, ensuring all nodes execute correctly and the final content reaches the target destination autonomously.

### 6.2. Technical Verification
*   **Unit Tests:** String sanitization and regex logic.
*   **Integration Tests:** File migration and folder structure validation.
*   **End-to-End (E2E):** Full MK-pipeline run against sample assets.
